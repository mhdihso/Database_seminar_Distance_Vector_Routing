# DuckDB and Recursive CTEs: The `USING KEY` Advantage

To understand why DuckDB is the perfect tool for implementing Distance-Vector Routing (DVR) in SQL, we first need to understand `WITH RECURSIVE` and how DuckDB extends it.

## What is a Recursive CTE?

A **CTE** (Common Table Expression) is simply a temporary named result set, defined by the `WITH` clause, that you can reference within a `SELECT`, `INSERT`, `UPDATE`, or `DELETE` statement.

A **Recursive CTE** (`WITH RECURSIVE`) is a CTE that references itself. It consists of two parts combined by a `UNION` or `UNION ALL`:

1. **Base Case:** Runs once at the beginning to populate the initial rows.
2. **Recursive Step:** Runs repeatedly. In standard SQL, this step reads the rows added in the *previous* iteration, performs some logic, and *appends* the new rows to the overall result table. The recursion stops when the recursive step returns exactly zero rows.

### The Problem with Standard SQL for Routing

In a standard SQL database like PostgreSQL or MySQL, Recursive CTEs are **append-only**. 

When a router in DVR discovers a new, cheaper path to a destination it already knew about, it needs to **update** its routing table. Standard recursive CTEs cannot update rows. Instead, they append the new path alongside the old path. 
To get the actual best paths in standard SQL, you must:
1. Let the algorithm find all possible acyclic paths.
2. Track a `visited_nodes` array to prevent infinite loops when traversing network cycles.
3. Apply a massive `GROUP BY ... MIN(cost)` query at the very end to filter out the suboptimal paths.

This is highly inefficient in terms of memory and runtime, and it doesn't accurately model how physical routers work.

---

## The DuckDB Solution: `USING KEY`

DuckDB introduces a powerful extension: **Recursive CTEs with `USING KEY`**.

This extension allows the CTE to break the "append-only" rule. It acts like a built-in Key-Value store or Dictionary for your recursive query.

```sql
WITH RECURSIVE routing_table(...) USING KEY (from_node, to_node) AS ...
```

By defining a primary key `(from_node, to_node)`, DuckDB guarantees that the final table will only ever contain **one row per key**. 

### How It Works

DuckDB gives you access to two different versions of your CTE during the recursive step:
1. `table_name`: Contains ONLY the rows generated by the **last iteration**.
2. `recurring.table_name`: Contains the **entire union table built so far**.

When you yield a row in the recursive step with a key that already exists, DuckDB **replaces** the existing row instead of appending to it.

### Why this is perfect for Distance-Vector Routing (DVR)

In the real world, DVR routers (like those using RIP or BGP) operate exactly like this:
1. They look at the updates they received in the last tick (`table_name`).
2. They compare the proposed new routes against their current, full routing table (`recurring.table_name`).
3. If the new route is cheaper, they overwrite the old route (`USING KEY`).
4. If they made any updates, they broadcast those specific updates to their neighbors in the next tick.

DuckDB allows us to write SQL that is a 1:1 conceptual match for the physical networking algorithm.
